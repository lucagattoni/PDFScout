## Goal

Enable `is_continued` flagging in the extraction pipeline and add two new tests:
- **E3** — verifies the extraction prompt correctly sets `is_continued=True` on a
  page-spanning paragraph fragment (extraction tier only)
- **F5** — verifies the hierarchy node correctly applies Rule 2 when a block has
  `is_continued=True` in the manifest (hierarchy tier only, pre-built blocks)

The E3 + F5 pair covers the full feature without coupling two concerns into one test.

The existing 28/28 e2e tests must continue to pass throughout.

---

## Problem statement

`is_continued` has three components, each with different status:

| Component | Status | Location |
|-----------|--------|----------|
| Schema field | Present in all three schemas — no description | `schemas/*.json` |
| Extraction instruction | **Missing** — prompt never mentions `is_continued` | `src/nodes/worker_node.py` |
| Hierarchy usage | Present — Rule 2 exists, but can never trigger | `src/nodes/hierarchy_node.py` |

Because the extraction prompt is silent on `is_continued`, Claude never sets it `true`.
Rule 2 ("If a block has `is_continued=true`, the first block of the next page is its
child") is dead code. Paragraphs split across pages appear as two unlinked sibling blocks.

### Why E3 and F5 are separate tests

`_run_e_test` (used by E1 and E2) mocks `hierarchy._call_api` with empty relations — all
blocks receive `parent_id=None`. Asserting Rule 2's `parent_id` output inside an E test
would always fail. The concerns are therefore split across test groups:

- **E3** (in `TestGroupE`): tests only that the extraction prompt sets `is_continued=True`
  on the page-1 fragment. Hierarchy is still mocked — only extraction matters here.
- **F5** (in `TestGroupF`): tests only that the hierarchy node correctly parents the
  continuation block when `is_continued=True` appears in the manifest. Uses pre-built
  blocks — no PDF, no extraction.

---

## What `is_continued` means in this pipeline

When a paragraph starts on page N and its text is cut off at the bottom — continuing
at the top of page N+1:

- **Page-N extraction**: the fragment block is returned with `is_continued=true`.
- **Page-(N+1) extraction**: the continuation block is a normal block; no special flag.
- **Hierarchy node Rule 2**: "If a block has `is_continued=true`, the first block of
  the next page is its child." → `continuation.parent_id = fragment.block_id`.

This is a **parent-child relationship**, not a merge. Two blocks remain in the output,
linked by `parent_id`.

The hierarchy node passes `is_continued` in the manifest:
```python
"is_continued": b.get("is_continued", False),
```
If Claude omits `is_continued` from a block dict, it defaults to `False` — schema
validation passes regardless since the field is not in `required`. The E3 test therefore
asserts `b.get("is_continued") is True` (explicit `True` required; absence = `False`).

---

## What is NOT in scope

- Merging continuation fragments into a single block (changes the output schema shape)
- Changing hierarchy Rule 2 semantics or the RELATION_TOOL definition
- Adding `is_continued` to `required` in the schema (optional by design)
- Asserting `is_continued=false` on non-continuation blocks (fragile)
- `is_continued` for list_item or table blocks that span pages
- End-to-end integration test of extraction → hierarchy in a single run (see Risk 7)

## Approach

Two sequential phases.

### Phase 1 — Schema description + extraction prompt update

**Entry condition**: `ANTHROPIC_API_KEY` available (regression tests require real API calls).

**Change 1 — `schemas/*.json`** (all three: baseline_core, invoice, scientific_paper):

Update `is_continued` from:
```json
"is_continued": { "type": "boolean", "default": false }
```
to:
```json
"is_continued": {
  "type": "boolean",
  "default": false,
  "description": "Set to true when this block's text is truncated at the bottom of the page and resumes at the top of the next page. Omit or set false for all complete blocks."
}
```

**Change 2 — `src/nodes/worker_node.py`**:

In the extraction prompt f-string, insert after "Coordinates must follow [ymin, xmin,
ymax, xmax] order." and before "Use the tool...":
```
"If a block's text is cut off at the bottom of the page and continues at the top of the
next page, set is_continued=true. Leave is_continued=false (or omit it) for all complete
blocks."
```

Both a positive constraint ("when to set true") and a negative constraint ("when NOT to
set true") are required. Without the negative constraint, Claude may defensively flag any
block near the bottom of a page. This instruction belongs in the base prompt, not in
`_doc_type_instructions`, since `is_continued` applies to all document types. For
single-page documents there is no "next page", so the instruction is a no-op.

**After changes — regression tests**:
```
uv run pytest tests/integration/ -m e2e -v
```
If any test fails, **revert both changes and document why** — do not adapt tests.

**After changes — E1/E2 spurious-flag inspection**:

A passing regression suite is necessary but not sufficient. The prompt addition could
silently cause Claude to set `is_continued=true` on a page-1 block in E1/E2 — E1/E2 do
not assert `is_continued` or `parent_id`, so this would not fail any test but would
degrade production quality. Inspect explicitly:

### Phase 2 — E3 + F5 tests

**Entry condition**: Phase 1 passes all 28 existing tests AND E1/E2 inspection shows no
spurious flags.

#### E3 — extraction-only assertion

**Fixture** (`grp_e_continuation.pdf`): A 2-page PDF with one block per page — a
paragraph split at the page break. One block per page eliminates ambiguity about which
block should have `is_continued=True` and which is "first" on page 2.

Both strings were measured at Helvetica 12pt and fit within the 170mm default cell width:

- **Page 1** — `draw_text` at `x=20mm, y=230mm` (near page bottom, A4=297mm):
  ```
  "The pipeline handles documents through extraction, classification, parsing, and"
  ```
  Ends with "and" — no period; sentence is incomplete. (Measured: 148.0mm ✓)

- **Page 2** — `draw_text` at `x=20mm, y=20mm` (top of page):
  ```
  "hierarchical assembly, making it suitable for production workloads."
  ```
  Begins with "hierarchical" (lowercase) — unambiguous continuation. (Measured: 124.2mm ✓)

Claude sees the full PDF when extracting each page. The content-level signal ("and" /
"hierarchical") makes the continuation evident without requiring visual layout inference.

**Generator**: Add `_make_continuation()` to `grp_e_multipage.py`; include
`grp_e_continuation.pdf` in `generate()`'s return list (registers in `manifest.json`).

**Determinism pre-check**: Before committing E3, run three times to verify the
extraction assertion is deterministic:
```
uv run pytest tests/integration/test_synthetic_grp_e.py::TestGroupE::test_e3_continuation -v
```
Require 3/3 passes.

**E3 test**:
```python
async def test_e3_continuation(self):
    """Paragraph split across pages: page-1 fragment has is_continued=True.
    Hierarchy is mocked (E-group convention); only extraction is asserted."""
    blocks = await _run_e_test(str(_PDFS / "grp_e_continuation.pdf"))

    page1_blocks = [b for b in blocks if b["bbox"]["page_number"] == 1]
    page2_blocks = [b for b in blocks if b["bbox"]["page_number"] == 2]
    assert page1_blocks, "No blocks found on page 1"
    assert page2_blocks, "No blocks found on page 2"

    continued = [b for b in page1_blocks if b.get("is_continued") is True]
    assert continued, (
        "No page-1 block has is_continued=True — extraction prompt instruction "
        "or schema description may be insufficient"
    )
    assert len(continued) == 1, (
        f"Expected exactly 1 is_continued=True block on page 1, got {len(continued)}"
    )
```

The `len(continued) == 1` assertion ensures the fixture is working as designed (one
continuation block, not multiple spuriously flagged blocks).

#### F5 — hierarchy Rule 2 narrow test

Rule 2 ("If a block has `is_continued=true`, the first block of the next page is its
child") has never been tested. F5 tests it in isolation using pre-built blocks.

**F5 fixture blocks** (defined in `grp_f_hierarchy.py`):

```python
# F5: [page-1 is_continued paragraph, page-2 continuation paragraph]
# Verifies hierarchy Rule 2: is_continued=True → first page-2 block is its child.
F5_BLOCKS = [
    {
        "block_id": "p1c",
        "type": "paragraph",
        "text": "This text is cut off at the bottom of page one, and",
        "bbox": {"page_number": 1, "coordinates": [280, 70, 295, 500]},
        "is_continued": True,
    },
    {
        "block_id": "p2c",
        "type": "paragraph",
        "text": "continues at the top of page two.",
        "bbox": {"page_number": 2, "coordinates": [20, 70, 35, 500]},
    },
]
```

`geometric_pre_sorter` order: p1c first (page=1, ymin=280), p2c second (page=2, ymin=20).
The manifest sent to the hierarchy LLM includes `is_continued: true` for p1c and the
text "cut off at the bottom of page one" as reinforcing context.

**F5 test**:
```python
async def test_f5_cross_page_continuation(self):
    """[page-1 is_continued para, page-2 para] → page-2 block is child of page-1 block
       (hierarchy Rule 2)."""
    from src.nodes.hierarchy_node import layout_hierarchy_agent_node

    result = await layout_hierarchy_agent_node(make_state(list(F5_BLOCKS)))
    blocks = result["hierarchical_document_tree"]["structured_payload"]
    by_id = {b["block_id"]: b for b in blocks}

    assert by_id["p1c"]["parent_id"] is None, (
        "Page-1 fragment should be root-level (no preceding heading)"
    )
    assert by_id["p2c"]["parent_id"] == "p1c", (
        f"Expected page-2 continuation parent_id='p1c', "
        f"got {by_id['p2c']['parent_id']!r} — hierarchy Rule 2 may not have triggered"
    )
```

**Determinism pre-check**: Run F5 three times before committing; require 3/3 passes.

**If F5 fails**: Rule 2 is not reliably honoured by the hierarchy LLM. Strengthen the
hierarchy prompt instruction — e.g., rewrite Rule 2 as a mandatory constraint: "When
`is_continued=true` appears for a block, you MUST assign the first block of the next
page as its child." Do not remove F5 or weaken the assertion.

## Open questions / risks

1. **Spurious `is_continued=true` on multi-page documents.** The prompt change could
   cause Claude to flag blocks near the bottom of any page as continued, even if text
   is complete. The negative constraint ("Leave is_continued=false for all complete
   blocks") mitigates this.
   _Mitigation_: The E1/E2 inspection step in Phase 1 is a concrete gate.

2. **Hierarchy Rule 2 non-determinism.** Even with `is_continued=true` in the manifest
   and Rule 2 explicitly stated, the LLM may not always honour it. If it returns empty
   relations, all `parent_id` values become `None` (orphan path), causing F5 to fail.
   _Mitigation_: The 3-run pre-check gates F5. If non-deterministic, rephrase Rule 2
   as a mandatory constraint in the hierarchy prompt.

3. **Claude may rewrite the extracted text for the page-1 fragment.** The E3 test does
   not check text content — only `is_continued` — so this does not affect correctness.

4. **`is_continued=true` on the last page of a document.** No "next page" exists; the
   flag would have no effect in the hierarchy (no page-N+1 block to parent). This is a
   low-risk edge case not covered by E3 and can be addressed in a follow-up if observed.

5. **`is_continued` is optional (`default: false`, not in `required`).** Claude can
   omit it entirely; `b.get("is_continued", False)` returns `False`. The E3 test
   correctly treats absence as failure via `b.get("is_continued") is True`.

6. **The E3 fixture's page-1 block at `y=230mm` leaves 67mm of whitespace below.** This
   is not a mid-paragraph page overflow but a deliberate placement. Claude sees whitespace
   below "and" and a page break, reinforcing the "cut off" signal alongside the
   content-level cue.

7. **No end-to-end integration test.** E3 verifies extraction; F5 verifies hierarchy.
   Neither tests the two working together in a single pipeline run (which is the
   production path). This gap exists because the E-group convention mocks the hierarchy.
   A follow-up could add an integration variant that does not mock the hierarchy.
