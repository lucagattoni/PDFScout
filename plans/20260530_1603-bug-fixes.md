# Bug Fix Plan — Devil's Advocate Review Findings
Date: 2026-05-30

## Background

A full devil's advocate review (`/code-review --effort max`) was run against the
Claude PDF Chat migration (implemented 2026-05-30). Eight confirmed or plausible
bugs were identified across six files. This plan describes each fix, the
rationale, and the execution order.

---

## Findings & Fixes

### 1 — CRITICAL · `src/state.py:10` — Retry accumulation corrupts the block buffer

**Problem.**  
`merge_flat_blocks` is a pure append reducer. Every time `window_parser_node`
retries page 1, it appends a fresh set of blocks to `extracted_flat_blocks`. By
the time the pioneer succeeds, the list contains 2–4 copies of page-1 content
with different `block_id` values (Claude generates opaque, unique IDs on every
call; there is no instruction to reuse prior IDs). The dedup guard in
`hierarchy_node` (`block_id` set membership) is therefore inert and all
duplicate copies pass through to the hierarchy agent, inflating the tree.

**Fix.**  
Replace the bare append reducer with one that is reset-aware: if the incoming
value is the sentinel `None`, clear the buffer; otherwise append as before.
Update `extractor_node` (see finding #2) and `retry_node` to emit `None` when a
reset is needed.

```python
# state.py
def merge_flat_blocks(left: list, right: list | None) -> list:
    if right is None:
        return []
    return left + right
```

### 2 — CRITICAL · `src/nodes/extractor_node.py:32` — Reset is swallowed by the reducer

**Problem.**  
`extractor_node` returns `"extracted_flat_blocks": []` intending to clear the
buffer before a new extraction run. Because `merge_flat_blocks` appends its
argument, `[] + [] = []` only works on the very first call. On checkpoint resume
the stale buffer from a prior run is never cleared.

**Fix.**  
Use the sentinel `None` (from finding #1's updated reducer) instead of `[]`:

```python
# extractor_node.py
return {
    ...
    "extracted_flat_blocks": None,   # signals reducer to clear
    "extraction_warnings": [],
}
```

Also update `retry_node` to emit `None` for `extracted_flat_blocks` so that each
retry attempt starts from a clean buffer rather than accumulating on top of the
previous failed attempt's blocks.

### 3 — HIGH · `src/nodes/worker_node.py:68` / `src/nodes/hierarchy_node.py:91` — Bare `next()` raises `RuntimeError`, bypasses tenacity

**Problem.**  
```python
tool_block = next(b for b in response.content if b.type == "tool_use")
```
If the API returns no `tool_use` block (contract violation, API error, or
model refusal), `next()` raises `StopIteration`. Inside an `async def`, PEP 479
converts that to `RuntimeError: coroutine raised StopIteration`. Tenacity is
decorating `_call_api`, not the enclosing coroutine, so this `RuntimeError`
propagates uncaught to the LangGraph graph runner, crashing the node with no
retry.

**Fix.**  
Use `next(..., None)` with an explicit guard:

```python
tool_block = next((b for b in response.content if b.type == "tool_use"), None)
if tool_block is None:
    raise ValueError(
        f"API returned no tool_use block for page {current_page}. "
        f"Content types: {[b.type for b in response.content]}"
    )
```

Apply the same pattern in `hierarchy_node.py:91`.

### 4 — HIGH · `src/nodes/retry_node.py:8` — Empty block list passes schema validation, wastes entire retry budget

**Problem.**  
When `active_blocks` is `[]`, `jsonschema.validate({"document_type": ..., "blocks": []}, schema)`
succeeds — none of the schemas have a `minItems` constraint on `blocks`. The
`except jsonschema.ValidationError` branch is never entered, `error_detail` is
set to the fallback `"Unknown schema violation."`, and all three retry attempts
send a useless, non-specific error message back to the model.

**Fix.**  
Add an explicit emptiness check before validation:

```python
# retry_node.py
if not active_blocks:
    error_detail = (
        f"No blocks were extracted for page {state['current_page']}. "
        "Return at least one block covering this page's content."
    )
else:
    try:
        jsonschema.validate({"document_type": ..., "blocks": active_blocks}, schema)
        error_detail = "Unknown schema violation."
    except jsonschema.ValidationError as e:
        error_detail = str(e.message)
```

### 5 — HIGH · `src/schema_registry.py:9` — Schema path resolved relative to process CWD

**Problem.**  
`Path("schemas")` resolves to the current working directory, not to the package
root. Running the application from a subdirectory, a test runner with a
different CWD, or as an installed package raises `FileNotFoundError`. The
fallback path has the same defect and is opened without an existence check.

**Fix.**  
Anchor the path to the file's location:

```python
# schema_registry.py
_SCHEMA_DIR = Path(__file__).parent.parent / "schemas"
```

### 6 — MEDIUM · `src/nodes/hierarchy_node.py:95` — Missing `block_id` in `relation_map` silently sets `parent_id=None`

**Problem.**  
`relation_map.get(block["block_id"])` returns `None` for any `block_id` the
hierarchy LLM omits from its `relations` output. The block is silently promoted
to root with `parent_id = None`. No warning is emitted and the tree structure is
wrong with no observable signal.

**Fix.**  
Emit a warning to `extraction_warnings` for every block that falls through:

```python
if block["block_id"] not in relation_map:
    warnings.append(
        f"block_id '{block['block_id']}' missing from relation_map; "
        "promoted to root."
    )
    block["parent_id"] = None
else:
    block["parent_id"] = relation_map[block["block_id"]]
```

Return the warnings in the node's output dict.

### 7 — MEDIUM · `src/nodes/worker_node.py:35` / `src/nodes/classifier_node.py:46` — Blocking file I/O inside async coroutines

**Problem.**  
`_encode_pdf` uses synchronous `open()` + `f.read()` inside `async def`
functions. For large PDFs (up to 32 MB per Claude's limit) this blocks the
entire event loop for the full read duration, defeating the concurrency model
and stalling all other in-flight coroutines.

**Fix.**  
Wrap the synchronous read with `asyncio.to_thread`:

```python
async def _encode_pdf(file_path: str) -> str:
    def _read() -> str:
        with open(file_path, "rb") as f:
            return base64.standard_b64encode(f.read()).decode("utf-8")
    return await asyncio.to_thread(_read)
```

Update both call sites to `await _encode_pdf(...)`. Extract to a shared
`src/utils/pdf_utils.py` to eliminate the current duplication between the two
nodes (see finding #8).

---

### 8 — LOW · `main.py:10` / `src/nodes/extractor_node.py:6` — Duplicate hash functions risk silent divergence

**Problem.**  
`_compute_hash` in `main.py` and `_hash_file` in `extractor_node.py` are
byte-for-byte identical. If one is updated (e.g., algorithm change, chunk-size
change) while the other is not, the `thread_id` used for checkpointing will no
longer match the `pdf_hash` stored in state, silently breaking resume/cache
lookups.

**Fix.**  
Extract to `src/utils/pdf_utils.py` (same module as the shared `_encode_pdf`
from finding #7), export `hash_file`, and import it in both `main.py` and
`extractor_node.py`.
